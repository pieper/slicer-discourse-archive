<h2><a name="p-132263-pip_install-features-pr-ebrahim-2" class="anchor" href="#p-132263-pip_install-features-pr-ebrahim-2" aria-label="Heading link"></a>pip_install features PR (Ebrahim)</h2>
<ul>
<li><a href="https://github.com/Slicer/Slicer/pull/9010">Slicer#9010</a></li>
<li>Docstrings shortened; most examples are moved to a <a href="https://slicer--9010.org.readthedocs.build/en/9010/developer_guide/script_repository.html#python-package-management">new script repository section</a>.</li>
<li>Split off the <code>slicer.util.pip_*</code> out of <code>slicer.util</code> into a new  package.
<ul>
<li>With the name <code>slicer.packaging</code> we run into a name conflict with the <code>packaging</code> module (a dependency of <code>pip</code>).</li>
<li>Landed on the name: <code>slicer.pydeps</code>. So <code>slicer.pydeps.pip_ensure(...)</code>, etc.
<ul>
<li><code>py</code> in the name of a python module is kind of annoying. Or is it? pydicom?</li>
<li>Maybe we should actually make <code>slicer.util</code> into an import package and make <code>slicer.util.pydeps</code> (with a better name than <code>pydeps</code>). Because we are importing all <code>slicer.*</code> things, and so if “sub-utilities” are all direclty under the <code>slicer</code> package then we prevent ourselves from using good package names due to this name shadowing issue. We may want to set the precedent now that “sub-utilities” should be <code>slicer.util.*</code>.</li>
<li>Review can proceed and when we finalize the naming and package organization we can do it at the end before merging.</li>
</ul>
</li>
</ul>
</li>
</ul>
