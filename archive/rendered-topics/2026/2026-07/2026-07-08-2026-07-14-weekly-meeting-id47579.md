<li><strong>When the target version is already installed and you haven’t set <code>SLICER_IF_EXISTING</code> env variable, what should a no-interaction run do: reinstall it (today’s no-terminal behavior) or exit 0 without touching it?</strong> This is the idempotency question — it decides whether repeated automation runs re-download ~1 GB or become no-ops.</li>
<li><strong>Do you also need quiet output (no logo, no progress bar) for logs, or is suppressing prompts enough?</strong> <em>(default: prompts only — logo and progress bars are already suppressed when output isn’t a terminal)</em></li>
<li><strong>Linux/macOS only, or do you automate the Windows installer too?</strong></li>
</ol>

<ol>
<li><strong>If installing dependencies fails, should the whole install fail (good for <code>RUN</code> steps in a Dockerfile) or warn and continue?</strong></li>
<li><strong>If the script is <em>not</em> root, should it attempt <code>sudo</code> (which may password-prompt — colliding with no-interaction mode) or refuse and print the command as it does today?</strong></li>
<li><strong>Do you run Slicer headless in the container (e.g. <code>xvfb-run Slicer --no-main-window</code>)?</strong> If so, should the mode optionally install headless extras like <code>xvfb</code>, or stay strictly at Slicer’s runtime library list?</li>

<li><strong>When run from an extracted tree, should it wire that tree up in place (wherever it was extracted), move it to the standard <code>~/.local/opt</code> location, or ask?</strong></li>
<li><strong>Should it be the same <code>install.sh</code> gaining an offline/local mode (one file, could be dropped into the tarball as-is), or a separate minimal script that contains only the wiring logic?</strong></li>
<li><strong>Must it work fully offline (no checksum/version lookups at all)?</strong></li>
<li><strong>Should it also offer uninstall of that copy?</strong> <em>(default: yes — the uninstall logic already exists in <code>uninstall_linux</code>)</em></li>
