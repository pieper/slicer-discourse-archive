<h2><a name="p-133298-extension-review-and-publishing-1" class="anchor" href="#p-133298-extension-review-and-publishing-1" aria-label="Heading link"></a>Extension review and publishing</h2>
<p><a href="https://github.com/Slicer/ExtensionsIndex/issues/2341">ExtensionsIndex#2341</a> by Steve: make it easy to install extension PRs, since unfunded review creates a backlog that hurts contributors and users.</p>
<h3><a name="p-133298-to-discuss-2" class="anchor" href="#p-133298-to-discuss-2" aria-label="Heading link"></a>To discuss:</h3>
<ul>
<li>Tier-based filtering of extensions (status update from Kyle).</li>
<li>Move packaging off the factory onto GitHub Actions:
<ul>
<li>Andras’s prototype: <a href="https://github.com/Slicer/Slicer/pull/9118">Slicer#9118</a>.</li>
<li>Example use: <a href="https://github.com/lassoan/SlicerMONAIAuto3DSeg/pull/105">SlicerMONAIAuto3DSeg#105</a>.</li>
</ul>
</li>
<li>Proposals as to how much review we want:
<ul>
<li><em>Andras:</em> allow publish-without-review; surface quality via ratings/likes; promote highly-rated ones into higher tiers.</li>
<li><em>Steve:</em> treat it like publishing a repo/website; minimal formal process; users recommend to users.</li>
<li><em>Murat:</em> keep peer review but make high-demand extensions pay an annual review/support fee; use an AI agent to screen new submissions for malicious code and ship others as “use at your own risk.”</li>
</ul>
</li>
<li>Sandboxing to protect users from untrusted extension code (and untrusted AI agents). Options raised: containers, cloud execution, and eventually browser-based (SlicerTrame, VTK wasm, pyodide).</li>
<li>A feature in Slicer to install/build extension PRs directly. Pure-Python extensions are low-hanging fruit; also a “check out and build” button when running a locally built Slicer.</li>
</ul>
