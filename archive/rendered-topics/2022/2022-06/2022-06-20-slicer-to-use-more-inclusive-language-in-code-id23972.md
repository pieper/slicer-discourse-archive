<p>I propose that on <span data-date="2022-06-23" class="discourse-local-date" data-timezone="America/New_York" data-email-preview="2022-06-23T04:00:00Z UTC">2022-06-23T04:00:00Z</span> Slicer integrates the following PR to adopt language to improve inclusivity in the code base. Juneteenth’s official date was yesterday (June 19th). Therefore this change in the Slicer code base is also a relevant DEI (Diversity, equity and inclusion) effort. DEI is not about a single day, but about continued action so the Slicer community should continue to host an inclusive environment not just in the forum, but also through how we code.</p>

Although it is often nice to maintain using current terminology, as it relates to improving inclusivity, this should be higher priority.

This PR includes acting upon what was originally described by @jcfr at https://discourse.slicer.org/t/2021-04-06-hangout/16953/3?u=jamesobutler which detailed:

&gt; Removal of divisive terminology regarding branch name associated with repository hosted in the Slicer and commontk GitHub organization. We will rename `master` to `main`

Renaming of the `master` branch to `main` had been indicated in April 2021, but has not yet been acted upon, until now...

GitHub has improved the ability to rename a branch (https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-branches-in-your-repository/renaming-a-branch) which makes it easy to switch to a new name with minimal consequences. This has to be completed by the admin of the Slicer repository (@jcfr).

<p><strong>What this means for you as a developer?</strong><br>
You will need to update your local environment with the reference to the new branch name of <code>main</code>. When you visit <a href="https://github.com/Slicer/Slicer" class="inline-onebox" rel="noopener nofollow ugc">GitHub - Slicer/Slicer: Multi-platform, free open source software for visualization and image computing.</a> for the first time after the change you will be shown instructions on how to update your local environment. These commands are also shown below.</p>
<pre><code class="lang-auto">git branch -m master main
git fetch origin
git branch -u origin/main main
git remote set-head origin -a
</code></pre>

## Post #6 by @lassoan (2022-06-23 20:41 UTC)

<p><strong>Instructions for developers for switching to the new main branch</strong></p>
<p>Instructions for command-line git users:</p>
<pre><code class="lang-auto">git branch -m master main
git fetch origin
git branch -u origin/main main
git remote set-head origin -a
</code></pre>
<p>Instructions for TortoiseGit users:</p>
<ul>
<li>Fetch → keep defaults, just click OK (this downloads all the branches from the remote <code>origin</code> repository)</li>
<li>Switch/Checkout → select Branch: remotes/origin/main → click OK (creates a <code>main</code> branch locally from the remote <code>origin</code>)</li>
</ul>
<p>If you had any changes in your local <code>master</code> branch then you can use the <code>cherry-pick</code> function of git to bring them over to the new <code>main</code> branch.</p>
