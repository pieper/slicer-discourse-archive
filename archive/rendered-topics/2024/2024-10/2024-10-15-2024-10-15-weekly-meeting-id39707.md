## Post #4 by @jcfr (2024-10-16 21:32 UTC)

<h3><a name="p-118449-meeting-notes-1" class="anchor" href="#p-118449-meeting-notes-1" aria-label="Heading link"></a>Meeting Notes</h3>
<h4><a name="p-118449-terminologies-2" class="anchor" href="#p-118449-terminologies-2" aria-label="Heading link"></a>Terminologies</h4>
<ul>
<li>
<p><strong>Module Improvements:</strong> Discussed potential enhancements to the Terminologies module. See the discussion here: <a href="https://github.com/Slicer/Slicer/issues/6975#issuecomment-2402467988">Slicer Issue #6975</a>.</p>
</li>
<li>
<p><strong>Terminology, Color Table, and Segment Relationship:</strong></p>
<ul>
<li>There is a significant disparity between the vast number of available terminology terms and the limited number of segments. The color table serves as a bridge:
<ul>
<li>Analogy: Terminology represents all available paint colors, the color table is the palette, and segments are the canvas.</li>
</ul>
</li>
</ul>
</li>
<li>
<p><strong>Handling Missing Terms:</strong></p>
<ul>
<li>It is not problematic if a segment lacks a corresponding term in the terminology. A streamlined editing approach could involve direct color table modifications. This would require an improved file format for color tables (details below).</li>
<li>Idea: Develop a new Terminologies extension to connect with an online repository of terminologies (e.g., on GitHub).</li>
</ul>
</li>
<li>
<p><strong>Color Table File Format Enhancements:</strong></p>
<ul>
<li>Transition to CSV format, ensuring compatibility with scene load and save operations.</li>
<li>Include all essential terminology information: name, code, coding scheme designator, category, modifier, and optional anatomical location.</li>
<li>Allow for JSON properties to be optional, omitting unused entries.</li>
</ul>
</li>
<li>
<p><strong>Displaying Color Tables in Terminology Contexts:</strong></p>
<ul>
<li>Limit to discrete options and filter based on specific attributes (e.g., “compatible”: adequate metadata).</li>
</ul>
</li>
<li>
<p><strong>Creating a “Short List” Color Table:</strong></p>
<ul>
<li>Enable users to generate a new color table based on current segments, providing a customizable starting point.</li>
</ul>
</li>
<li>
<p><strong>MorphoDepo (MorphoCloud) Requirements:</strong></p>
<ul>
<li>Each user receives a unique color table tied to their segmentation.</li>
<li>Consider features and ideas from the <a href="https://github.com/cpinter/SlicerSegmentationVerification">SegmentationVerification extension</a>.</li>
</ul>
</li>
<li>
<p><strong>Using Uberon in Terminologies:</strong></p>
<ul>
<li>Hierarchical structure required, such as categories (e.g., “skeletal system / spine”) to create a more organized tree structure, replacing the current flat list.</li>
</ul>
</li>
<li>
<p><strong>Next Steps:</strong></p>
<ul>
<li>Enhance color tables to store metadata effectively.</li>
<li>Integrate color tables within the terminology selector for streamlined user experience.</li>
</ul>
</li>
</ul>
<h4><a name="p-118449-slicer-trame-3" class="anchor" href="#p-118449-slicer-trame-3" aria-label="Heading link"></a>Slicer &amp; Trame</h4>
<ul>
<li><strong>Trame Integration:</strong>
<ul>
<li>
<p>Discussed the integration of <a href="https://kitware.github.io/trame/">trame</a> with Slicer to support backend GPU rendering.</p>
</li>
<li>
<p>Render window content is converted into a compressed JPEG image, which is transmitted to the front end as a web application.</p>
</li>
<li>
<p>Transitioning to in-render window widgets would minimize dependencies on Qt.</p>
</li>
<li>
<p>Refactoring will be required to enable segmentation editor effects to function independently from Qt.</p>
</li>
<li>
<p>Progress related to Slicer refactoring is ongoing; see <a href="https://github.com/AlexyPellegrini/Slicer/commits/slicer-as-a-lib" class="inline-onebox">Commits · AlexyPellegrini/Slicer · GitHub</a></p>
</li>
<li>
<p>On-going development of trame-based Slicer should be released open-source in the first quarter of 2025. For questions, reach out to <a class="mention" href="/u/finetjul">@finetjul</a>, <a class="mention" href="/u/thibault_pelletier">@Thibault_Pelletier</a> and <a class="mention" href="/u/jcfr">@jcfr</a></p>
