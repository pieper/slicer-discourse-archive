<h4><a name="p-115880-slicer-tutorials-4" class="anchor" href="#p-115880-slicer-tutorials-4" aria-label="Heading link"></a>Slicer Tutorials</h4>
<ul>
<li>
<p><strong>References</strong></p>
<ul>
<li><a href="https://github.com/Slicer/Slicer/issues/6676">Issue #6676</a>: Improve Slicer training portal.</li>
<li><a href="https://github.com/Slicer/Slicer/pull/7910">PR #7910</a>: DOC: Add link to new tutorials.</li>
</ul>
</li>
<li>
<p><strong>Tutorial Organization</strong></p>
<ul>
<li>Organize tutorials into top-level categories (e.g., SlicerIGT).</li>
<li>Use one repository per tutorial to facilitate creation, revision control, storage of PDFs, and sample data as release assets. In the future, CI will automate translation and test execution.</li>
</ul>
</li>
<li>
<p><strong>Tutorial Naming Convention</strong></p>
<ul>
<li>Assign unique, descriptive names for each tutorial to avoid confusion with similar titles. Consider using IDs similar to SlicerIGT, for example:
<ul>
<li><code>STC-SEG-101</code> (Slicer Training Compendium - Segmentation - Beginner)</li>
<li>Examples: <code>STC-GEN-101</code>, <code>STC-SEG-101</code>, <code>STC-REG-101</code>, <code>STC-NEURO-102</code>, <code>STC-IGT-USR-102</code>, <code>STC-IGT-DEV-102</code>, <code>STC-DEV-101</code>, <code>STC-MORPH-101</code>.</li>
</ul>
</li>
<li>These IDs will uniquely identify tutorials, and the tutorial finder could use these IDs as reference.</li>
</ul>
</li>
<li>
<p><strong>Next Steps</strong></p>
<ul>
<li><a class="mention" href="/u/lassoan">@lassoan</a> : Draft the main page content based on existing tutorials - <a href="https://hackmd.io/@zpX1d6ofQ1i7_ArZsJ3_-g/r17QDhE20">Draft Link</a>.</li>
<li><a class="mention" href="/u/jcfr">@jcfr</a>: Rename <code>tutorials.slicer.org</code> GitHub project to <code>training.slicer.org</code> <img src="https://emoji.discourse-cdn.com/twitter/check_mark.png?v=15" title=":check_mark:" class="emoji" alt=":check_mark:" loading="lazy" width="20" height="20">.</li>
<li><a class="mention" href="/u/lassoan">@lassoan</a>: Ask <a class="mention" href="/u/cpinter">@cpinter</a> to create a dedicated repository for the segmentation tutorial.</li>
</ul>
</li>
<li>
<p><strong>Proposed GitHub Topic</strong></p>
<ul>
<li>Use the topic: <code>3d-slicer-training</code> for related repositories.</li>
</ul>
</li>
</ul>
<h4><a name="p-115880-slicer-reusable-github-actions-and-workflows-5" class="anchor" href="#p-115880-slicer-reusable-github-actions-and-workflows-5" aria-label="Heading link"></a>Slicer Reusable GitHub Actions and Workflows</h4>
<ul>
<li>Following Slicer <a href="https://github.com/Slicer/Slicer/pull/7900">PR#7900</a>, the GitHub workflow <a href="https://github.com/Slicer/Slicer/blob/main/.github/workflows/commit-message.yml"><code>commit-message.yml</code></a> is now re-usable.</li>
<li>Maintaining this workflow within the <code>Slicer/Slicer</code> project prevents Dependabot from effectively tracking updates specific to the pinned workflow, as it instead suggests updates referencing all Slicer changes rather than those specific to the workflow.</li>
<li><strong>Next:</strong> Create dedicated Slicer repositories to organize each GitHub reusable workflow or action.</li>
</ul>
