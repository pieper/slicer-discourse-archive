<p>It is great if you can reproduce this! We got similar complaints a few times but the users who reported could never do a thorough investigation.</p>
<p>Maybe it is due to girder not implementing some less frequently used HTTP GET features correctly. For example, <a href="https://discourse.slicer.org/t/incomplete-slicer-package-download-gzip-stdin-invalid-compressed-data-format-violated/29748/12">girder does not implement file chunk download correctly</a> (therefore resuming an interrupted download results in a corrupted file).</p>
<p>It could be interesting to see if download speed depends on time of day, depends on what software you use, and it could be interesting to have a look at the HTTP GET request to see if there is anything different from the browser in that jetstream2 instance compared to the browser on your computer.</p>

<p>If geogram Boolean mesh operations are robust that’s amazing news, because the toolkit has permissive license, so it is not problem to use it Slicer.</p>
<p>As a short-term solution, you can vendorize the library (copy source code into Sandbox), but it is slightly nicer if you can build it as a CMake ExternalProject. If the code proves to work well then it would make sense to add a VTK remote module for it.</p>
<p>The .ply extension is not a problem at all, you can specify what extension you support in the .xml file (see example <a href="https://github.com/Slicer/SlicerSurfaceToolbox/tree/master/Decimation">here</a>).</p>
