## Post #3 by @jcfr (2024-12-06 16:53 UTC)

<h3><a name="p-120232-meeting-notes-1" class="anchor" href="#p-120232-meeting-notes-1" aria-label="Heading link"></a>Meeting Notes</h3>
<h4><a name="p-120232-curved-planar-reformation-cpr-2" class="anchor" href="#p-120232-curved-planar-reformation-cpr-2" aria-label="Heading link"></a>Curved Planar Reformation (CPR)</h4>
<ul>
<li>
<p><strong>VTK-Centric Approach</strong>:</p>
<ul>
<li><a class="mention" href="/u/thibault_pelletier">@Thibault_Pelletier</a> explained that the earlier VTK-centric approach implemented in <a href="https://gitlab.kitware.com/vtk/vtk/-/merge_requests/10766">MR-10766</a> was overly complex, leading to its abandonment.</li>
<li>He provided further details on the polydata-based approach currently under development by <a class="mention" href="/u/lucasgandel">@LucasGandel</a> .</li>
</ul>
</li>
<li>
<p><strong>Activiz Blog Post</strong>:</p>
<ul>
<li><a class="mention" href="/u/lucasgandel">@LucasGandel</a>  published a blog post about the Activiz release, highlighting a new mapper (<code>vtkOpenGLSurfaceProbeVolumeMapper</code>) implementing a polydata-based approach. See <a href="https://gitlab.kitware.com/LucasGandelKitware/vtk/-/tree/probe-volume-mapper?ref_type=heads">LucasGandelKitware/vtk@probe-volume-mapper</a></li>
<li>Read the post: <a href="https://www.kitware.com/activiz-9-4-curved-planar-reformation/">Activiz 9.4 - Curved Planar Reformation</a>.</li>
</ul>
</li>
<li>
<p><strong>Coordinate ID Concept</strong>:</p>
<ul>
<li>As noted in previous discussions, adding a “coordinate ID” could enable filtering nodes for display in specific views:
<ul>
<li><code>FrameOfReferenceID</code> (or <code>CoordinateSystemID</code>) was added to <code>vtkMRMLAbstractViewNode</code>.</li>
<li>If specified, the view displays only MRML data nodes with matching <code>FrameOfReferenceID</code>.</li>
</ul>
</li>
</ul>
</li>
<li>
<p><strong>Commonality with Resectograms</strong>:</p>
<ul>
<li>This approach aligns with techniques used in <strong>Slicer-Liver</strong>’s resectograms project. Learn more: <a href="https://projectweek.na-mic.org/PW38_2023_GranCanaria/Projects/SlicerLiver/">Slicer-Liver Project</a>.</li>
</ul>
</li>
<li>
<p><strong>New View for “Straighten” Functionality</strong>:</p>
<ul>
<li><a class="mention" href="/u/thibault_pelletier">@Thibault_Pelletier</a> suggested creating a new node derived from <code>vtkMRMLAbstractViewNode</code> with a dedicated displayable manager using a <code>vtkMRMLCameraWidget</code>.</li>
<li>Meshes (models) could be displayed with a dedicated overlay renderer.</li>
</ul>

<h3><a name="p-120232-discussions-after-sofa-week-for-slicer-sofa-3" class="anchor" href="#p-120232-discussions-after-sofa-week-for-slicer-sofa-3" aria-label="Heading link"></a>Discussions after SOFA Week for Slicer-SOFA</h3>
<ul>
<li>
<p><strong>Slicer-SOFA as a Platform for SOFA Plugins</strong>:</p>
<ul>
<li><strong>Slicer-SOFA</strong> has been developed with Slicer users in mind, but <strong>SOFA users</strong> often create their own plugins that they may want to use within Slicer.</li>
<li>Discussed enabling developers to integrate their SOFA plugins into Slicer effectively.</li>
</ul>
</li>
<li>
<p><strong>Proposed Solution</strong>:</p>
<ul>
<li>Wrap SOFA plugins as Slicer extensions that leverage the SOFA libraries deployed by Slicer-SOFA.</li>
<li><strong>Slicer</strong>’s existing CMake infrastructure can facilitate finding the required libraries for SOFA plugins.</li>
</ul>
</li>
</ul>
<h3><a name="p-120232-discussion-about-dynamicmodeler-4" class="anchor" href="#p-120232-discussion-about-dynamicmodeler-4" aria-label="Heading link"></a>Discussion about DynamicModeler</h3>
<ul>
<li>
<p><strong>Extrusion Tool Enhancements</strong>:</p>
<ul>
<li><a class="mention" href="/u/mau_igna_06">@mau_igna_06</a> will submit a pull request to add a combobox for fixed or scaling factor mode when using line or point markups. Plane behavior will remain unchanged.</li>
<li>Discussed potential for rotational extrusion and the utility of <code>vtkRuledSurfaceFilter</code>.</li>
</ul>
</li>
<li>
<p><strong>DynamicModeler Sources Tool</strong>:</p>
<ul>
<li>Proposed adding tools to create shapes such as spheres, cubes, cylinders, and text as dynamic sources.</li>
</ul>
</li>
<li>
<p><strong>Downsampling Models and Adding Sources</strong>:</p>
<ul>
<li>Suggested a DynamicModeler tool to downsample points on a model and place a source model (e.g., using <code>vtkGlyph3D</code>) at each point.</li>
<li>Downsampling techniques could include:
<ul>
<li><strong>Uniform remeshing</strong></li>
<li><strong>vtkPoissonDiskSampler</strong></li>
<li><strong>vtkMaskPoints</strong></li>
<li>Example: <a href="https://examples.vtk.org/site/Cxx/VisualizationAlgorithms/SpikeFran/">SpikeFran Visualization</a>.</li>
</ul>
</li>
</ul>
</li>
<li>
<p><strong>Dynamic Meshes and Markups</strong>:</p>
<ul>
<li>For meshes with dynamic shapes and positions/orientations, implementing this functionality as a custom “markups” may be preferable.</li>
<li>Consider exploring existing features in the <strong>SlicerExtraMarkups</strong> extension.</li>
</ul>
</li>
</ul>
