## Post #4 by @jcfr (2025-07-01 14:09 UTC)

<h3><a name="p-126431-meeting-notes-1" class="anchor" href="#p-126431-meeting-notes-1" aria-label="Heading link"></a>Meeting Notes</h3>
<h4><a name="p-126431-experiment-with-rpyc-for-streaming-scalar-fields-2" class="anchor" href="#p-126431-experiment-with-rpyc-for-streaming-scalar-fields-2" aria-label="Heading link"></a>Experiment with RPyC for Streaming Scalar Fields</h4>
<ul>
<li>
<p><a class="mention" href="/u/pieper">@pieper</a> explored the use of <a href="https://rpyc.readthedocs.io/en/latest/">RPyC</a> (Remote Python Call) to stream <strong>scalar fields</strong> associated with <strong>unstructured grids</strong>.</p>
</li>
<li>
<p><strong>What is RPyC?</strong><br>
A Python library for <strong>symmetrical remote procedure calls</strong>, <strong>clustering</strong>, and <strong>distributed computing</strong>. It can use <strong>shared memory</strong> for efficient local communication of large arrays between processes.</p>
</li>
<li>
<p><strong>Current Bottleneck</strong>:</p>
<ul>
<li>When updating the scalar field, <strong>the surface mesh is re-extracted during each render</strong>, even though the underlying unstructured grid remains unchanged.</li>
<li>This causes performance issues for large meshes.</li>
</ul>
</li>
<li>
<p><strong>Next Steps</strong>:</p>
<ul>
<li>Investigate possible <strong>rendering optimizations</strong>, likely at the <strong><code>vtkMRMLModelDisplayNode</code></strong> level, to avoid redundant surface extraction.</li>
</ul>
</li>
</ul>
