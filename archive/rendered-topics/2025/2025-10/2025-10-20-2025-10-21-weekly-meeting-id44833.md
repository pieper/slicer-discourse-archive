## Post #2 by @RafaelPalomar (2025-10-21 11:52 UTC)

<p>I’m currently trying to create a custom visualization based on surface models where I need my own mappers (for custom shaders). It looks to me that this route would require a custom implementation of <code>MRMLModelNode</code>, <code>MRMLModelDisplayNode</code> and <code>MRMLModelDisplayableManager</code>.</p>
<p>Due to the way <code>MRMLModelDisplayableManager</code> filters nodes, it is not possible to simply inherit from <code>MRMLModelDisplayNode</code>, as any derived class will be processed by <code>MRMLModelDisplayableManager</code>.</p>
<p>It would be great if we can find some time to discuss this. Maybe we can adjust the filtering of DMs so inherited classes can be handled by custom DMs?</p>

## Post #4 by @jcfr (2025-10-28 13:40 UTC)

<h3><a name="p-129498-meeting-notes-1" class="anchor" href="#p-129498-meeting-notes-1" aria-label="Heading link"></a>Meeting Notes</h3>
<h4><a name="p-129498-slicer-510-2" class="anchor" href="#p-129498-slicer-510-2" aria-label="Heading link"></a>Slicer 5.10</h4>
<ul>
<li>Target date for tagging the <strong>Slicer 5.10</strong> release is first week of November .</li>
<li>This release will be the <strong>final version based on Qt 5</strong>, ahead of the transition to Qt 6.</li>
</ul>
<h4><a name="p-129498-lightbox-removal-3" class="anchor" href="#p-129498-lightbox-removal-3" aria-label="Heading link"></a>LightBox Removal</h4>
<ul>
<li>Discussed <a href="https://github.com/Slicer/Slicer/pull/8776">PR-8776</a>:<br>
<em><code>ENH: Simplify Slice view removing LightBox feature</code></em>.</li>
<li><a class="mention" href="/u/thibault_pelletier">@Thibault_Pelletier</a> will assist with finalizing the PR.</li>
<li>This is part of a broader effort to streamline rendering by removing legacy LightBox support.</li>
</ul>
<h4><a name="p-129498-extending-mrmldisplaynodes-and-mrmldisplayablemanagers-4" class="anchor" href="#p-129498-extending-mrmldisplaynodes-and-mrmldisplayablemanagers-4" aria-label="Heading link"></a>Extending <code>MRMLDisplayNodes</code> and <code>MRMLDisplayableManagers</code></h4>
<ul>
<li>Use case discussed: Creating a <strong>custom 3D View</strong> in the side panel to show custom model visualizations using <strong>custom mappers</strong> (similar to the <em>Resectogram</em> idea from <a href="https://projectweek.na-mic.org/PW38_2023_GranCanaria/Projects/SlicerLiver/">PW38 SlicerLiver project</a>).</li>
</ul>
<p><strong>Key technical considerations:</strong></p>
<ul>
<li>Custom mappers require <strong>custom MRMLDisplayableManagers</strong>, which must be implemented in <strong>C++</strong>.</li>
<li>While extending <code>vtkMRMLModelDisplayNode</code> is tempting for surface models, inherited managers (like <code>vtkMRMLModelDisplayableManager</code>) will process <strong>all subclasses</strong>, which may not be desirable.</li>
<li>One solution is to derive from <code>vtkMRMLDisplayNode</code> directly for full control.</li>
<li>Interaction customization (e.g., changing 3D view controls) can be done via scripting:<br>
<a href="https://slicer.readthedocs.io/en/latest/developer_guide/script_repository.html#make-the-3d-view-rotate-using-right-click-and-drag">Example from documentation</a>.</li>
<li><a class="mention" href="/u/thibault_pelletier">@Thibault_Pelletier</a> referenced the <a href="https://github.com/KitwareMedical/SlicerLayerDisplayableManager">KitwareMedical/SlicerLayerDisplayableManager</a> module, now integrated in Slicer 5.9 nightlies, as a helpful reference for creating custom rendering pipelines in both <strong>C++ and Python</strong>.</li>
</ul>
<h4><a name="p-129498-unity-based-vr-app-for-dentistry-5" class="anchor" href="#p-129498-unity-based-vr-app-for-dentistry-5" aria-label="Heading link"></a>Unity-based VR App for Dentistry</h4>
<ul>
<li><em>Alex Myers</em> introduced his Unity-based VR application focused on dental workflows.</li>
<li>Utilizes <strong>Slicer’s TotalSegmentator extension</strong> to generate anatomical models for use in the app.</li>
</ul>
<p><strong>Discussion points:</strong></p>
<ul>
<li>
<p>Challenges arise due to the <strong>lack of direct access to scientific toolkits</strong> (e.g., PyDICOM, SciPy, ITK, VTK) when using Unity.</p>
</li>
<li>
<p>To make the project more relevant for the Slicer community:</p>
<ul>
<li>A <strong>free-tier licensing option</strong> would be beneficial.</li>
<li>A <strong>bridge between Unity and Slicer</strong> would help streamline data transfer and reproducibility.</li>
</ul>
</li>
</ul>
